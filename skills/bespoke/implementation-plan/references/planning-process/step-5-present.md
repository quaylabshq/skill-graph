# Step 5: Present

**Status: MANDATORY — never skip this step.**

Present the completed plan to the user and wait for explicit approval before any implementation begins.

## Presentation Format

1. **Lead with a brief summary** — 2-3 sentences covering what the plan achieves and the approach
2. **Show the full plan** using the output format (core template + adaptive sections)
3. **Include the verification report** — from [step 4.5](step-4.5-verification.md), showing which agents verified the plan, their verdicts, and any residual minor issues
4. **Highlight key decisions** — explicitly call out choices you made and why
5. **Flag risks and uncertainties** — anything that might change during implementation
6. **Ask for explicit approval** — "Does this plan look right? Should I adjust anything before proceeding?"

## Handling Feedback

| Feedback Type | Action |
|---------------|--------|
| Approval | Begin implementation following the plan |
| Minor adjustments | Update the plan and re-present the changed sections |
| Major rethink | Return to the relevant earlier step (clarification, research, or decomposition) |
| Partial approval | Get clarity on which parts are approved and which need rework |

## Approval Is Required

Never auto-execute after planning. The plan is a proposal, not a command. The user must explicitly approve before implementation begins.

## Post-Approval Behavior

Once approved:

1. Use the plan as a structured todo list for implementation
2. Follow the step ordering from the plan
3. If implementation reveals something the plan didn't account for, pause and surface it to the user rather than deviating silently

## Rules

1. Always present the plan — never skip to implementation
2. Ask specific questions about gaps — not generic "does this look good?"
3. If feedback contradicts earlier answers, surface the contradiction
4. Approval stands for the scope presented — do not expand scope after approval
